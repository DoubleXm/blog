>[!INFO]
> 一组操作的集合（多条 `sql` 语句的集合），这个集合是一个不可分割的工作单位；
>
> 会将集合内的所有操作一起提交，如果提交过程中有任何一条 `sql` 出现错误，便回退到整个事务执行之前的操作；
>
> 要么同时成功，要么同时失败；

## 事务的提交方式

```sql
-- 只会对当前会话生效
select @@autocommit; -- 默认是 1 表示事务默认提 交
set @@autocommit = 0; -- 表示不会自动提交
```

## 事务的回滚与提交

```sql
-- 通过 commit 提交事务, rollback 回滚事务
commit;
rollback;
```

## 转账示例

假设用户 a 给用户 b 转账；需要先扣除 a 的余额然后给 b 增加余额；

```sql
create table account (id int primary key auto_increment, name varchar(16), balance int);
insert into account(name, balance) values ('a', 500), ('b', 1000);

-- 扣除 a 的余额
update account set balance = balance - 100 where name = 'a';
-- 增加 b 的余额
update account set balance = balance + 100 where name = 'b';
-- 查询所有的余额
select * from account;
```

由于事务是会默认提交的，所以要提前关闭 `set @@autocommit = 0;`

在没有出错的情况下，上述的 `sql` 执行完毕后，最后增加 `commit;` 就会提交事务，数据会完全被修改；

当执行的过程中出现错误时则需要手动执行 `rollback;` 回滚事务，让被修改的数据还原；

## 开启事务

在不修改事务提交方式的前提下，可以显示的开启事务；在要被执行的操作前增加以下语句

```sql
-- 开启事务
start transaction;
-- 或者
begin;

-- 扣除 a 的余额
update account set balance = balance - 100 where name = 'a';
-- 增加 b 的余额
update account set balance = balance + 100 where name = 'b';
-- 查询所有的余额
select * from account;

commit;
-- 或者，sql 自上而下执行，失败时自然不会运行到这里，需要手动执行一下 rollback 进行回滚
-- 从 coding 的逻辑中，可以放在 catch 中去执行 rollback
rollback;
```

## 事务的四大特性

- 原子性 Atomicity： 事务是不可分割的最小操作单元；
- 一致性 Consisteny：事务完成的时候，必须许所有数据都保持一致的状态；
- 隔离性 Isolation：在并发的时候，A 事务在操作的时候，它不会影响 B 事务的执行；反之亦然；
- 持久性 Durability：事务不管是提交还是回滚，它对数据库中数据的改变都是永久的。

## 并发事务问题

在事务没有提交之前数据时在缓冲池中操作的，当多个事务访问相同的数据时会出现各种竞态和不同步的问题；

### 脏读、不可重复读、幻读

- 脏读：一个事务读取到了另外一个事务还没有提交的数据；
- 不可重复读：一个事务先后读取同一条记录，但两次读取到的数据不同；

*当事务 A 查询数据数据时为结果 1，当第二次再查询的时候，期间的数据被事务 B 给修改了，导致了结果的变更；*

- 幻读：一个事务查询某一条记录的时候，没有对应的记录，但是在插入记录的时候，又发现这条记录已经存在了；

*其实和上面的场景类似，事务 A 查询的时候查不到数据，插入的时候又被事务 B 给插入了，但是也没有 commit，导致事务 A 查不到数据又插入不了数据。*

### 事务的隔离级别

| 隔离级别 | Read Uncommitted | Read Committed | Repeatable Read 默认 | Serializable |
| ------------- | :-----------: | :-----------: | :-----------: | :-------------: | 
| 脏读 | ✅ | ❌ | ❌ | ❌ |
| 不可重复读 |✅ | ✅ | ❌ | ❌ |
| 幻读 |✅ | ✅ | ✅ | ❌ |

脏读不一定是 bug，都有存在的意义，比如：在一个事务中插入大量的数据，要很久，这时候问你插入了多少条数据，就可以通过脏读去看已经插入的数据量；

```sql
-- 查看事务的隔离级别
select @@transaction_isolation;

-- 设置事务的隔离级别
set {session | global} transaction isolation level <隔离级别>;
-- 示例
set session transaction isolation level read committed;
```

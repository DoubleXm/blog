## 通识

从技术视角看，大模型本质上是一个基于参数的概率模型。它不会逐字保存训练资料，而是把训练数据中的语言结构、知识关联和推理模式压缩到参数中；推理时再根据当前上下文，计算下一个 token 的概率分布。

大语言模型通常写作 `LLM`，全称是 `Large Language Model`。它是一类以自然语言和代码为主要建模对象的深度学习模型。

大语言模型能力主要来自三件事：

- 算法：主流 LLM 基于 `Transformer`，核心机制是 `attention`。token 会先映射成向量，再经过多层权重矩阵和 attention 计算，动态建立上下文中不同 token 之间的依赖关系。
- 数据：训练语料决定模型能学习到哪些语言模式、事实关联、代码结构和表达风格。数据质量、覆盖范围和清洗策略，通常比单纯堆数据量更关键。
- 算力：训练阶段依赖大量 GPU/加速卡反复更新参数；推理阶段则不断基于上下文计算下一个 token。参数越多、上下文越长，通常意味着更高的显存、延迟和调用成本。

核心概念：

- 参数：模型内部可学习的权重数字。`10M ≈ 1,000 万`，`1B ≈ 10 亿`，`10B ≈ 100 亿`，`1T ≈ 1 万亿`。参数多通常代表容量更大，但不等于事实一定更新、推理一定更强。
- token：模型读写文本的最小计费/计算单位，可能是一个字、一个词，也可能是半个词。粗略看，中文 `1 字 ≈ 1 token`，英文 `1 个词 ≈ 1~2 token`。
- token 规模：`1K ≈ 1,000 token`，`10K ≈ 1 万 token`，`1M ≈ 100 万 token`。输入、输出、工具结果、历史对话都会占 token；输出越长，消耗越多。
- 向量：多维数字坐标，用来表示 token 的含义和特征。比如“北京”和“上海”的坐标会更接近，“北京”和“香蕉”会更远；如果做搜索，会用坐标距离找相近内容，但大模型生成回答时，主要是把这些向量继续交给参数权重和 attention 计算。
- 权重：模型训练后留下来的大量数字，可以理解成被压缩进模型里的经验规律。比如看到“北京是中国的”，权重会让“首都”比“水果”更可能出现。
- attention：注意力机制。在计算当前 token 时，判断上下文里哪些 token 更值得参考。比如“苹果发布了新手机”，模型会更关注“发布”“新手机”，而不是把“苹果”优先理解成水果。
- 上下文：这次请求里模型能看到的全部内容。比如一次会话里的系统提示、前面所有问答、当前问题、工具返回结果、文件片段，以及应用主动塞回来的思考摘要或执行记录，都可以成为上下文；没放进上下文的内容，模型这次就看不到，所以模型默认是没有记忆的。
- 上下文窗口：模型一次能容纳的上下文大小。超过窗口后，应用通常只能截断、摘要或丢掉一部分旧内容；被丢掉的内容模型无法继续参考，所以会表现得像“忘了前文、变笨了”。
- 缓存命中：多次请求前缀相同，服务商可能复用已计算的上下文。命中时通常更快、更便宜；前面内容一变，缓存就容易失效。
- Agent：基于 LLM 组织出来的执行系统。它会让模型在循环中理解目标、拆分步骤、调用工具或 API、观察结果，再继续决定下一步；可以理解为 `LLM + 工具 + 状态/记忆 + 任务规划`。如果说 LLM  是大脑，那么 Agent 就是手和脚。

一次对话的流程如下：

1. 应用把系统提示、历史对话、用户输入和工具说明拼成上下文。
2. 模型把文本切成 token，再把 token 转成内部向量。
3. `Transformer` 用参数权重和 attention 计算 token 之间的关系。
4. 模型得到“下一个 token”的概率分布，选出一个 token。
5. 把选出的 token 拼回上下文，重复第 3-5 步，直到回答结束、触发工具调用或达到长度限制。

![大模型逐个 token 生成回复流程](/ai/model-token-generation.svg)

常见生成参数会影响“怎么选下一个 token”：

- `temperature`：温度越低，越偏向选择高概率 token，回答更稳定、更保守；温度越高，低概率候选也更容易被选中，回答更多样，但也更容易跑偏。
- `top_p`：只在累计概率达到指定比例的候选 token 里抽样，常用于控制随机范围。
- `top_k`：只在概率最高的前 `K` 个候选 token 里抽样，`K` 越小越保守。
- `max_tokens`：限制最多生成多少 token，决定回复长度上限。
- `frequency_penalty`：降低已经频繁出现过的 token 概率，减少重复表达。
- `presence_penalty`：只要某类内容出现过，就降低它继续出现的概率，鼓励模型引入新内容。

## LangChain

是一个用来开发智能体工程（Agent Engineering）的框架，同时兼容 `py | ts` 两种语言；支持各种模型的调用；内部包含四个核心模块

- LangChain: 用于快速构建智能体，可兼容任何模型提供商
- LangGraph: 从底层一步步控制智能体的构建，包括记忆（Memory）、人机协同（HITL）等
- DeepAgents: 用于构建复杂的、处理多步骤的任务的智能体
- LangSmith: 用于测试、观察、评估、部署智能体

### 模型和 agent 创建

默认情况下 langchain 会自动读取环境变量中的 api key， 所以通过 `create_agent` 不需要指定 key， 但是变量名要符合官网规范 `OPENAI_API_KEY`


```py
# uv add langchain
from langchain.agents import create_agent

agent = create_agent(
    model="openai:gpt-5.5",
    system_prompt="你是一名天气助手",
)
result = agent.invoke(
    {
        "messages": [
            {"role": "user", "content": "今天天气怎么样"},
        ]
    }
)
print(result["messages"][-1].content_blocks)
```

但此种方式仅支持一些比较出名的厂商，像阿里云百炼等并不会提供，需要使用 `init_chat_model` 去创建一个 model，但是需要安装 `langchain-deepseek`

支持的所有外部模型列表 [model list](https://docs.langchain.com/oss/python/integrations/providers/overview)

创建出来的 model 可以直接丢给 `create_agent` 去使用

```py
import os
from langchain.chat_models import init_chat_model
from app.core.config import settings

DEEPSEEK_API_KEY = os.environ["DEEPSEEK_API_KEY"] = settings.DEEPSEEK_API_KEY

model = init_chat_model(
    model="deepseek-v4-flash",
    # 通过后续的选项转发到 openai
    model_provider="openai",
    api_key=settings.DEEPSEEK_API_KEY,
    base_url="https://api.deepseek.com",
)
# 阻塞
result = model.invoke("你是谁")
# 流
result = model.strame("你是谁")
```

`init_chat_model` 支持的关于模型常用参数包括

- max_retries 最大重试次数，默认 6
- timeout 超时等待时间 s
- max_tokens 最大响应 token 数
- temperature 控制模型随机性，越高越具有创造性，越低越确定性，取值范围 0 - 2 默认 1
- top_p 模型会考虑前 top_p 概率的 token 的结果。所以 0.1 就意味着只有包括在最高 10% 概率中的 token 会被考虑。默认是 1

完整调用

```py
import os
from langchain.agents import create_agent
from langchain.chat_models import init_chat_model
from app.core.config import settings

DEEPSEEK_API_KEY = os.environ["DEEPSEEK_API_KEY"] = settings.DEEPSEEK_API_KEY

model = init_chat_model(
    model="deepseek-v4-flash",
    model_provider="openai",
    api_key=settings.DEEPSEEK_API_KEY,
    base_url="https://api.deepseek.com",
    temperature=0.8,
    max_tokens=4000,
    top_p=1,
    timeout=20,
    max_retries=6,
)

agent = create_agent(
    model,
    system_prompt="你是七龙珠动漫中的克林",
)
results = agent.stream({"messages": [{"role": "user", "content": "我是 18 号"}]}, stream_mode="messages")

for chunk, _ in results:
    if chunk.content:
        print(chunk.content, end="", flush=True)
```

## 工具

是 `create_agents` 的一个参数，在最开始的时候，agent 会将上下文和工具包装好交给模型，然后模型分析需要使用哪个工具，在让 agent 执行工具，最后模型分析好在返回给用户

```py
from langchain.agents import create_agent
from langchain.chat_models import init_chat_model
from langchain.messages import HumanMessage
from langchain.tools import tool

from app.core.config import settings


@tool
def get_weather(location: str) -> None:
    """天气查询工具"""
    return f"{location} 今天是晴天"


model = init_chat_model(
    model="deepseek-v4-flash",
    model_provider="openai",
    base_url="https://api.deepseek.com",
    api_key=settings.DEEPSEEK_API_KEY,
)
agent = create_agent(model, tools=[get_weather])

result = agent.invoke({"messages": [HumanMessage("上海今天的天气怎么样")]})

for message in result["messages"]:
    print(message.pretty_print())

"""
================================ Human Message =================================

上海今天的天气怎么样
None
================================== Ai Message ==================================
Tool Calls:
  get_weather (call_00_9LXySUHiZlOC4Vq4r5EY8414)
 Call ID: call_00_9LXySUHiZlOC4Vq4r5EY8414
  Args:
    location: 上海
None
================================= Tool Message =================================
Name: get_weather

上海 今天是晴天
None
================================== Ai Message ==================================

上海今天**天气晴朗**，是个大晴天！☀️

如果您需要更详细的天气信息（如温度、风力等），请随时告诉我，我可以进一步帮您查询。
None
"""
```

工具的定义默认会使用工具函数的 doc 注释、参数、返回值作为 ai 识别的依据， 也可以直接给 `@tool` 传入参数

```py
@tool("获取天气", description="根据传入的地区获取天气")
def get_weather(location: str) -> None:
    return f"{location} 今天是晴天"
```

参数比较多的时候，可以使用 `pydantic` 做类型校验

```py
class WeatherParams(BaseModel):
    location: str
    days: int

@tool(args_schema=WeatherParams)
def get_weather(location: str, days: int) -> str:
    """获取天气"""
    return f"未来 {days} 天 {location} 晴天"
```

## 结构化输出

实测发现不能使用思考模型，也有可能姿势存在问题；

- 通过 prompt 约定必须返回 json 格式的对象，但是随着 temperature 的调整会使输出变得不确定
- response_format 基于 pydantic 做约定使模型返回一定符合规范

```py
class Res(BaseModel):
    message: str = Field(description="返回数据")
    code: int = Field(description="业务自定义 code")

agent = create_agent(
    model,
    tools=[get_weather],
    response_format=Res,
)
response = agent.invoke({"messages": [HumanMessage("未来几天北京天气怎么样")]})
# 通过 structured_response 获取到结构化的数据
print(response["structured_response"].model_dump_json())
```

## 记忆

模型的对话默认是没有记忆的，记忆分为两种，长期记忆和短期记忆

- 长期记忆：跨任务跨会话内容共享一般用户知识库等，需要用到向量库存储数据
- 短期记忆：只存在与当前会话的上下文，新开会话就会丢失，上下文过长模型就会变笨

langchain 中为短期记忆提供了几种存储方案，基于内存或者基于数据库存储；

在运行时 langchain 会将会话（Messages）作为 AgentState 的一部分，通过 CheckPointer 进行存储，每一次的对话存储一次，最终变成完整的会话历史

为了区分不同的会话记忆，每个会话要有自己的 thread_id，向 agent 发起会话的时候必须指定 thread_id

![alt text](/ai/check-pointer.png)

### 内存记忆

### 数据库记忆

长期存储支持 `langgraph-checkpoint-sqlite | langgraph-checkpoint-postgres | langgraph-checkpoint-mongodb` 等 [查看](https://docs.langchain.com/oss/python/integrations/checkpointers/index) 都需要进行安装


### 记忆管理策略 删除、修剪

使用 before_model 或者 after_model 在模型调用之前或者模型返回内容之后删除消息

```py
from langchain.agents import create_agent, AgentState
from langchain.agents.middleware import before_model, after_model
from langchain.messages import RemoveMessage
from langgraph.graph.message import REMOVE_ALL_MESSAGES
from langgraph.runtime import Runtime

@before_model
def trim_message(state: AgentState, runtime: Runtime) -> dict[str, Any] | None:
    messages = state["messages"]
    if len(messages) <= 2:
        return None

    # 删除所有的 messages
    return {"messages": [RemoveMessage(REMOVE_ALL_MESSAGES)]}

@after_model
def del_old_message(state: AgentState, runtime: Runtime) -> dict | None:
      messages = state["messages"]
      if len(messages) > 2:
          # 一个一个删
          return {"messages": [RemoveMessage(id=message.id) for message in messages[:2]]}

create_agent(model, middleware=[del_old_message, trim_message])
```

### 记忆管理策略 总结

删除和修剪实际上都会让模型丢失会话信息，总结则是通过另一个模型把历史消息进行总结；

```py
agent = create_agent(
    model,
    system_prompt="你是七龙珠中的破坏神比鲁斯。正在和孙悟空超级赛亚人之神的形态准备展开战斗",
    checkpointer=sqlite_checkpointer,
    # tools=[destroy],
    # middleware=[trim_message],
    middleware=[
        SummarizationMiddleware(
            model=init_chat_model(
                model="deepseek-chat",
                model_provider="openai",
                base_url="https://api.deepseek.com",
                api_key=settings.DEEPSEEK_API_KEY,
            ),
            trigger=("messages", 2),  # 超过 3 条时触发压缩
            keep=("messages", 1),  # 保留的会话数，超过两条
            # api_key=settings.DEEPSEEK_API_KEY,
            # base_url="https://api.deepseek.com",
        )
    ],
)
```

[常用选项](https://docs.langchain.com/oss/python/langchain/middleware/built-in#summarization)

trigger: 会话摘要的触发时机，有三种设置 and 关系，都要满足

- fraction (float): 模型上下文大小的比例（0-1）
- tokens (int): 令牌数量
- messages (int): 消息数量

keep：是指触发摘要后要保留的消息 and 关系，都要满足

- fraction (float): 要保留的消息占模型上下文大小的比例（0-1）
- tokens (int): 要保留的消息的令牌数量
- messages (int): 要保留的消息数量

- 项目内安装 `uv add "langgraph-cli[inmem]"`
- 通过 `langgraph new path/to/your/app --template new-langgraph-project-python` 会创建出来一个新的项目，如果是老项目改造可以直接增加一个 `langgrap.json` [配置文档](https://docs.langchain.com/oss/python/langgraph/application-structure#configuration-file)
- 环境变量中要至少包含 `LANGSMITH_API_KEY | LANGSMITH_PROJECT` 这两个变量

```json
// langgraph.json
{
  // 项目的依赖
  "dependencies": ["."],
  "graphs": {
    // [Agent文件路径]:[Agent变量名]
    "my_agent": "./your_package/your_file.py:agent"
  },
  // 环境变量
  "env": ".env"
}
```

**使用 LangSmith 调试 agent 时不能再指定 checkpointer 否则会报错的，因为 LangSmith 会帮助创建 checkpointer， 可以考虑使用环境变量隔离开生产和开发的情况**

```py
from uuid import uuid4

from dotenv import load_dotenv
from langchain.agents import create_agent
from langchain.chat_models import init_chat_model
from langchain_core.runnables import RunnableConfig
from langchain_tavily import TavilySearch

from app.core.config import settings

system_prompt = """
你是一名私人厨师。收到用户提供的食材照片或清单后，请按以下流程操作：
1.识别和评估食材：若用户提供照片，首先辨识所有可见食材。基于食材的外观状态，评估其新鲜度与可用量，整理出一份“当前可用食材清单”。
2.智能食谱检索：优先调用 web_search 工具，以“可用食材清单”为核心关键词，查找可行菜谱。
3.多维度评估与排序：从营养价值和制作难度两个维度对检索到的候选食谱进行量化打分，并根据得分排序，制作简单且营养丰富的排名靠前。
4.结构化方案输出：把排序后的食谱整理为一份结构清晰的建议报告，要包含食谱信息、得分、推荐理由、食谱的参考图片，帮助用户快速做出决策。

请严格按照流程，优先调用 web_search 工具搜索食谱，搜索不到的情况下才能自己发挥。
"""

config: RunnableConfig = {"configurable": {"thread_id": uuid4()}}

model = init_chat_model(
    model="kimi-k2.6",
    model_provider="openai",
    base_url="https://api.moonshot.cn/v1",
    api_key=settings.MOONSHOT_API_KEY,
)

# TavilySearch 需要从环境变量中加载
load_dotenv()
# 这是一个 tool，用来做 web search
tavily_search = TavilySearch(max_results=5, topic="general")

# 和环境变量中 : 后面的值要保持一致
agent = create_agent(
    model,
    system_prompt=system_prompt,
    tools=[tavily_search],
)
```
